# 开发规范

本文档是《云辰盾_开发规范文档.docx》的工程落地版。所有代码、接口、表结构、页面路由、提交记录必须遵守本规范。

## 1. 目录结构

- 按业务板块分目录，10 个板块对应 10 个模块目录。
- 前后端分离，多端目录独立：`backend/api`、`backend/data-center`、`frontend/pc-admin`、`frontend/miniapp`。
- 公共常量放 `shared/constants`，通用工具放 `shared/utils`，设计 Token 放 `shared/components`。
- 标准模板统一放 `templates`，按业务类型分目录。

## 2. 命名规范

- 接口路由以板块缩写开头：`/api/fs`、`/api/hr`、`/api/fin`。
- 数据库表名使用模块前缀：`fs_home_report`、`hr_employee`、`fin_charge`。
- 角色、权限、审批状态使用枚举或常量，禁止数字硬编码。
- 文件头部或类注释应说明所属板块、功能、创建日期、作者。

## 3. 权限系统

- 使用 RBAC：角色、资源、操作、数据范围四层解耦。
- 后端每个接口必须做角色和数据权限校验，不能只依赖前端菜单隐藏。
- 所有写操作必须记录操作日志，包含操作人、时间、模块、资源、改前改后数据。

## 4. 数据中台

- 模块间数据交互走数据中台接口，不允许模块直接跨库查询。
- 学生 ID 是全平台核心主键，成长档案、缴费、保险、心理、家校记录均以学生 ID 聚合。
- 指标计算、预警、排名、画像统一在数据中台完成，前端只取结果。

## 5. 审批流

- 人事流转、薪资审批、退费申请、离职交接等共用统一流程引擎。
- 审批状态统一为：`DRAFT`、`PENDING`、`PROCESSING`、`APPROVED`、`REJECTED`、`CANCELED`。
- 审批流转只发领域事件，消息推送由消息模块处理。

## 6. 前端规范

- PC 后台与小程序共用设计 Token：颜色、字体、间距、圆角、状态色。
- 每个业务板块对应独立路由分组。
- 列表、搜索、分页、表单、详情、审批页组件化。
- 数据大屏只调用数据中台聚合接口。

## 7. Git 规范

- `main`：稳定上线版本。
- `develop`：日常开发主分支。
- `feature/板块名-功能`：功能开发分支。
- 提交格式：`[板块] 动作: 描述`，例如 `[FS] feat: add home report workflow`。
- 每个阶段完成后打 Tag：`v1.0-phase1`。

## 8. 文档规范

- 每个业务板块必须有 README，写清功能边界、角色、依赖、表、接口。
- 接口文档由 Swagger / Knife4j 生成，接口变更必须同步注释。
- 数据库变更必须走 Flyway 迁移脚本。

## 9. 安全与隐私

- 心理、薪资、家庭隐私、身份证号、联系方式等敏感字段加密存储。
- 日志必须脱敏，禁止记录明文密码、身份证号、完整手机号、心理原始记录。
- 文件上传限制类型、大小和访问权限，下载必须鉴权。
